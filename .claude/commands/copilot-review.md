You are automating the full Copilot review → fix → push → CI loop for the current branch's PR. Work through the phases below in order.

## Phase 1: Find the PR and Copilot Comments

1. Get the current branch:
   ```
   git branch --show-current
   ```
2. Find the open PR for this branch:
   ```
   gh pr view --json number,url,headRefName,title
   ```
   If no PR exists, stop and tell the user.

3. Fetch all review comments on the PR, filtering for those authored by `github-copilot[bot]` (or `copilot-pull-request-reviewer[bot]`):
   ```
   gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --paginate \
     --jq '[.[] | select(.user.login | test("copilot"; "i")) | {file: .path, line: .original_line, diff_hunk: .diff_hunk, body: .body}]'
   ```
   Also fetch top-level PR review bodies (non-inline):
   ```
   gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --paginate \
     --jq '[.[] | select(.user.login | test("copilot"; "i")) | {body: .body, state: .state}]'
   ```
   Combine inline comments and review bodies into a single list.

4. If no Copilot comments exist, tell the user "No Copilot review comments found on this PR." and stop.

5. Format the collected comments for the next phase. For each inline comment use:
   ```
   - <file_path>
   Comment on line +<line>
   """
   <body>
   """
   ```
   For top-level review bodies, include them as general notes at the top.

## Phase 2: Evaluate and Apply

Run a code-review evaluation on the collected comments using the project conventions in `CLAUDE.md`:

1. **Parse & Read** — read each referenced file and its context (±30 lines around the referenced line).
2. **Evaluate** — for each comment determine VALID / INVALID / DEFER per project conventions in `CLAUDE.md`. Apply these project-specific gates:
   - A suggestion that asks you to import a framework (`fastapi`, `sqlalchemy`, `redis`, `celery`, `pydantic`, `asyncpg`) into `src/domain/` or `src/application/` is INVALID — it breaks the hexagonal rule in `CLAUDE.md` → "Architecture — Hexagonal / Ports & Adapters".
   - A suggestion to turn a domain dataclass into a Pydantic model is INVALID — `CLAUDE.md` → "Code conventions" requires Pydantic only for API schemas.
   - A suggestion to introduce `Any` into `domain/` or `application/` is INVALID — `CLAUDE.md` → "Code conventions".
   - A suggestion to replace a `match` statement on actions with an if/elif chain is INVALID — `CLAUDE.md` → "Code conventions" prefers `match`.
   - A suggestion to add `print` is INVALID — use `structlog` per `CLAUDE.md`.
   - A suggestion to make async code sync (or vice-versa) should be evaluated against `CLAUDE.md` → "Async all the way down" before accepting.
3. **Report** — present a summary table:
   ```
   | # | File | Line | Verdict | Reason |
   ```
4. **Ask the user for confirmation** before applying any changes. The user may skip items or adjust approach.
5. **Apply** approved changes, then run the relevant local checks:
   - Python changes: `pytest tests/unit -x -q` (unit tier — must be fast per `CLAUDE.md`), and `mypy src` / `ruff check src` if configured.
   - Frontend changes (React/Vite under the frontend source tree): `pnpm tsc --noEmit` and `pnpm lint` if those scripts exist in `package.json`; otherwise skip.
   - If the change touches an adapter with integration tests, also run the matching `tests/integration/` subset.

If no comments are VALID, or the user skips all of them, skip Phase 3 (no push needed).

## Phase 3: Push Changes (if fixes were applied)

If at least one fix was applied:

1. Stage and commit:
   ```
   git add <changed files>
   git commit -m "fix: apply Copilot review suggestions"
   ```
   Use Conventional Commits per `CLAUDE.md` → "Code conventions" (`feat:` / `fix:` / `chore:`). Include a Co-Authored-By trailer.
2. Push to remote:
   ```
   git push
   ```
3. Tell the user the commit SHA and that the push is done.

## Phase 4: Check CI / Pipeline Status

After pushing (or immediately if no push was needed), poll the PR's CI status:

1. Wait a few seconds for checks to register, then:
   ```
   gh pr checks
   ```
2. If all checks pass — report success and stop.
3. If any check is **pending**, wait up to 3 minutes (poll every 30 seconds with `gh pr checks`) then re-evaluate.
4. If any check **fails**:
   - Fetch the failure logs:
     ```
     gh run view <run_id> --log-failed
     ```
   - Diagnose the failure. Common causes in this repo:
     - `pytest` failures → read the failing test output and fix the code (not the test, unless the test is wrong). Domain tests must stay I/O-free per `CLAUDE.md`.
     - `mypy` errors → fix types; never silence with `# type: ignore` unless the root cause is a third-party stub gap.
     - `ruff` / `black` formatting → run the formatter locally and recommit.
     - `alembic` migration check → verify migrations are additive and autogenerated heads aren't duplicated.
     - Docker / `docker compose` build failure on CI → check `requirements/*.txt` pinning.
     - Frontend: `pnpm tsc --noEmit` or `pnpm lint` failures → fix and recommit.
   - Apply fixes, then **commit and push again** (go back to Phase 3 for the new commit).
   - Re-check CI. Repeat up to **3 fix cycles**. If still failing after 3 cycles, stop and summarize what remains broken for the user to investigate manually.

## Rules

- **Never force-push** (`git push --force`). Always create new commits.
- **Never skip hooks** (`--no-verify`). Fix the root cause.
- **Never modify `BOARD.md` or `CLAUDE.md`** in this flow — this skill is CI-scoped only.
- **Never cross the hexagonal boundary** to make a Copilot suggestion work. If the only way to apply a suggestion is to import a framework into `domain/` or `application/`, reject the suggestion and note it in the report.
- **Always confirm before applying code changes** (Phase 2 step 4). The push and fix cycles (Phases 3–4) are automatic after user approval.
- **Respect project conventions.** If a Copilot suggestion conflicts with `CLAUDE.md` decisions, reject it and explain why.
- **Don't over-fix.** Only address what Copilot flagged plus what CI reports. Don't refactor surrounding code.
