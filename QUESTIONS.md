## Phase 1 — Domain core

### Player (task 1.2)
- [x] Starting HP? → **20**.
- [x] Which base stats and starting values? → **HP, Attack (start 3), Defense (start 1)**. Speed/Luck deferred to v2 (see Backlog in BOARD.md).
- [x] Inventory capacity — fixed slot count, weight-based, or unbounded? → **No general inventory / bag.** Items live in dedicated equipment slots only (see next answer).
- [x] Equipment slots (weapon / armor / accessory) or one generic bag? → **Dedicated slots only**: `weapon`, `armor`, `shield`, and a `consumables` stack (potions, cap 5 for v1). Picking up when a slot is full replaces the current item; the old item drops on the tile.
- [x] Level / XP system in v1, or flat stats with gear-only growth? → **No XP.** Flat stats, gear-only growth.

### Enemy + BehaviourType (task 1.3)
- [x] HP / damage scaling across floors — linear, exponential, hand-authored tables? → **Linear** for v1. Predictable TTK, easier to balance; the scaling function can be swapped later without touching the `Enemy` dataclass. Floor-1 melee damage cap of 1 HP/hit still holds.
- [x] Ranged enemies: which tile types block line-of-sight? (walls only? doors?) → **`WALL` and closed `DOOR` block LOS.** Rule: "solid, IRL view-blocking" tiles block. `FLOOR` and `STAIRS` do not. Future blockers (smoke, tall grass, pillars) slot in as new tile types without changing the rule.
- [x] Boss cadence — every 5th floor (per Backlog in BOARD.md) or something else for v1? → **Every 5th floor** (floors 5, 10, 15, …), matching the Backlog entry.
- [x] Does `BehaviourType` start as `MELEE | RANGED | BOSS` exactly, or more granular (e.g. `MELEE_AGGRESSIVE`, `MELEE_COWARD`)? → **`MELEE | RANGED | BOSS`** for v1, matching CLAUDE.md's Key Domain Concepts verbatim. Granular variants (`MELEE_COWARD`, etc.) are cheap to add in v2 — nothing else in the domain pattern-matches on the enum yet.

### Item + ItemType (task 1.4)
- [x] Which item types ship in v1? (potions, weapons, armor, scrolls, gold, keys?) → **`WEAPON | ARMOR | SHIELD | POTION | KEY`**. Each maps to an existing Player slot: weapon/armor/shield to their dedicated slots; POTION and KEY both live in the `consumables` stack (cap 5 total). Scrolls / gold deferred to v2.
- [x] Stackable items — flat count on one instance, or many instances? → **Flat `count: int = 1` on the `Item` instance.** Stackable types (`POTION`, `KEY`) merge into one instance on pickup; non-stackable types (`WEAPON`, `ARMOR`, `SHIELD`) always have `count=1`. Consumables stack is a list of ≤5 `Item` instances.
- [x] Item multiplier for scoring — per-type weight, per-rarity, or per-item value? → **Per-type weight.** Static `ItemType → float` map owned by `ScoreService` (exact values TBD during task 1.7). Rarity / per-item overrides can layer in later without breaking the contract.

### Floor (task 1.5)
- [x] Grid dimensions — fixed (e.g. 80×50) or scales with floor index? → **`80×50` fixed for v1.** Classic roguelike standard (Rogue, NetHack); enough room for 6–10 BSP rooms. May scale with floor index later — revisit once `DungeonGenerator` exists.
- [x] One down-staircase per floor, or multiple branches? → **One down-staircase per floor.** Linear descent for v1; branching deferred.
- [x] Do items and enemies live on the `Floor` or in separate collections keyed by position? → **Mixed, deliberately**: `enemies: list[Enemy]` (quiz-aligned — `Enemy.position` is intrinsic, so no key needed; ~5–20 enemies per floor makes O(n) lookup a non-issue for v1) and `items: dict[tuple[int, int], list[Item]]` (necessary — `Item` has no intrinsic position, so the dict key is canonical; list value supports stacking). Reverses the earlier dict-for-both answer after the [QUIZZES.md Q4 design intent](QUIZZES.md#L60) was surfaced. Worth a `DECISIONS.md` entry once `GameService` starts moving items between `Floor.items` and `Player`'s slots — that's where the asymmetry will bite.

### Dungeon (task 1.6)
- [x] Total depth — fixed (e.g. 20 floors) or endless? → **100 floors, fixed** for v1. Finite depth gives a concrete "win" state and caps run length for leaderboard fairness; endless mode deferred.
- [x] Seed source — client-provided, server-random, or both (daily seed mode)? → **Both.** Server-random for normal runs (fresh seed per `StartGame`); client-provided for daily/weekly leaderboard modes where every player gets the same static seed for fair comparison. The `seed: int` field stays the same shape — only the caller picks.
- [x] Is the player ref stored on `Dungeon` or passed separately to services? → **Passed separately** (Option B). `Dungeon` holds `dungeon_id`, `seed`, `floors`, `current_floor_index` — no `player` field. Services take both: `process_turn(dungeon, player, action)`. Accepted trade-off: more plumbing per signature now, but Player stays "who the user is" (future profile/unlocks persistent across runs) while Dungeon stays "this specific run" — co-op in v2 is additive, not a refactor. **Note:** this deviates from the [QUIZZES.md Task 1.6 Q3](QUIZZES.md#L69) hint ("Dungeon contains a Player") — the quiz wording may need updating, or the Q3 answer will accept that v1 chose the non-hinted shape deliberately.

### Score (task 1.7)
- [x] Exact formula — literally `floors × kills × item_multiplier`, or weighted (e.g. `floors² × kills × item_multiplier`)? → **Weighted on floors.** Depth dominates so descending is strictly better than grinding shallow floors. Exponent fixed at `floors_reached**2` (locked by ADR 0002 in `DECISIONS.md`).
- [x] Penalties for damage taken, turns used, or deaths? → **Damage-taken penalty** only. Rewards careful play; discourages tank-and-grind. Requires a cumulative `damage_taken` counter on the run state (likely `Player` or `Dungeon` — exact home TBD during implementation). Turns-used penalty deferred; deaths is a no-op in v1 (permadeath = one death per run).
- [x] Minimum floor for a score to count at all? → **Kill-based threshold** instead of floor-based: a run must have **≥ 5 kills** to qualify for the leaderboard (tentative — tune after playtesting). Measures actual engagement rather than just walking. Where the check lives is TBD: likely in `SubmitScore` (Phase 3) or the leaderboard query, not in `ScoreService.compute()` — keeps domain flexible.

### TileType (task 1.8)
- [x] Tiles beyond `WALL | FLOOR | STAIRS | DOOR`? (trap, water, chest, altar?) → **No additional tiles for v1.** Ship `WALL | FLOOR | STAIRS | DOOR` only — matches CLAUDE.md "Key Domain Concepts". `StrEnum` makes future variants (`TRAP`, `WATER`, `CHEST`, `ALTAR`) additive without breaking existing pattern matches.

### Action (task 1.9)
- [x] Actions beyond Move/Attack/UseItem/Descend/Abandon? Likely candidates: `Wait`, `PickUp`, `DropItem`, `Open` (door), `Equip`. → **Ship v1 with `Move | Attack | UseItem | Descend | Abandon | Wait | PickUp | Open`.** `Wait` (skip turn — needed in turn-based combat for regen/enemy-advance), `PickUp` (items live in `Floor.items` dict per the 1.5 decision; explicit pickup lets the player step over without auto-grabbing into a full slot), `Open` (closed `DOOR` blocks LOS per the 1.3 decision, so opening must be a discrete action). **Defer** `DropItem` (no general bag — slots auto-drop on replace per the 1.2 decision) and `Equip` (pickup goes straight into the slot, so no separate equip step in v1).
- [x] Is `Attack` implicit on `Move into enemy tile`, or a separate explicit action? → **Implicit.** Moving into an enemy-occupied tile resolves as an attack; no separate `Attack` action variant needs a target field. Keeps the input surface minimal (movement keys do double duty) and matches classic roguelike feel (Rogue, NetHack, DCSS). The `Attack` variant still exists in the union for future ranged/targeted actions and for explicit attack-without-move (e.g. attacking a diagonal tile from a position where moving there is blocked).

### DungeonGenerator (task 1.13)
- [x] BSP parameters — min room size, max recursion depth, corridor style (L-shape / straight)? → **Min room 4×4, max recursion depth 5, L-shape corridors.** 4×4 is the smallest room that admits a 1-tile-wide corridor entry on each side without degenerating into a single-tile cell. Depth 5 caps leaf rooms at 32, comfortably above the 6–10 target rooms for an 80×50 floor — picking a tighter cap risks under-partitioning, a looser one fragments the map. L-shape corridors (two straight segments meeting at a right angle) are the classic roguelike read on a tile grid and survive the inevitable case where BSP partitions don't line up; pure-straight corridors would fail too often given non-aligned partitions. The exact min-size and depth values are tunable knobs on `DungeonGenerator` — **not** baked-in constants — so playtesting can adjust without an algorithm rewrite.
- [x] Post-gen reachability check — enforce or trust the algorithm? → **Enforce** via flood-fill from the spawn tile after generation. BSP only *probabilistically* yields full connectivity once corridors are stitched in — bugs in the corridor pass produce orphan rooms that are nightmarish to debug post-hoc (player gets stuck, can't find stairs, leaderboard runs invalidated). Flood-fill is O(W·H) ≈ 4 000 ops on an 80×50 grid — negligible — and stays pure, so it's trivially seeded-equality testable. If unreachable tiles are detected, the generator should re-roll (with a bumped sub-seed) rather than silently patching, so the failure mode is "regen takes a few extra ms" rather than "weird half-broken floor."
- [x] Enemy / item placement — part of this generator or a separate step? → **Separate step**: `DungeonGenerator` produces pure geometry (`Floor` with tiles + stairs position only), and a downstream `populate_floor(floor, depth, rng) -> Floor` function adds enemies and items. This keeps the generator's contract narrow — "given a seed and floor index, produce a deterministic layout" — and makes seeded-equality testing of *layout* possible without entangling enemy/item RNG draws. It also decouples balance tuning (spawn density curves, drop tables, boss cadence) from algorithmic changes to the layout — you can re-tune Phase 4 of v1 without touching the BSP pass. Both functions take the same RNG (or split sub-RNGs from the parent seed) so the run remains fully reproducible.

### EnemyAI (task 1.15)
- [x] Line-of-sight gating, or blind Manhattan pathing? → **LOS-gated** for v1. Blind Manhattan pathing produces enemies that "know" the player's position through walls — visually wrong and feels unfair (the kind of thing players notice within five turns). LOS gating is the standard roguelike contract and aligns with the [task 1.3 decision](#enemy--behaviourtype-task-13) that `WALL` and closed `DOOR` block line of sight, so the same predicate that gates ranged attacks gates AI awareness — one rule, one mental model. Stealth-around-corners falls out of this for free.
- [x] FOV algorithm if LOS is needed — shadowcasting, raycasting, symmetric-shadowcasting? → **Symmetric shadowcasting.** Modern roguelikedev consensus pick (NetHack/DCSS-class projects use it). Two properties that matter: (1) **symmetric** — if the AI sees the player, the player sees the AI, so the rendering layer and the AI layer can share a single FOV pass without "I see them but they don't see me" bugs that break combat fairness; (2) **artifact-free at corners** — classic recursive shadowcasting has known corner-leak bugs, Bresenham raycasting has the asymmetry bug. Implementation is ~80 lines of pure code; lives in `domain/services/fov.py` so it stays adapter-free.
- [x] Wake-up radius — enemies chase always, or only when within N tiles? → **Within 8 tiles AND in LOS.** Combined predicate: `chebyshev_distance(enemy, player) <= 8 AND has_los(enemy, player)`. 8 is roguelike-standard (NetHack-ish) and corresponds roughly to one small-room diameter in Chebyshev distance — meaning enemies wake when the player enters their room/corridor segment but not from across the whole floor. The LOS half rewards stealth approach (slipping past a corridor while staying behind a wall), the radius half prevents enemies from chasing across the entire 80×50 grid the moment the player has any sightline. Once awoken, an enemy stays awoken for the rest of the floor — losing aggro after one missed turn would be too lenient and would cause oscillating chase/sleep behaviour at the LOS boundary.

### GameService.process_turn() (task 1.16)
- [x] State mutation model — does `process_turn` mutate `Dungeon` in place, or return a new `Dungeon`? → **Mutate in place.** The whole domain has already committed to mutability: the `Floor` docstring is explicit (*"Mutable by design. ... Mutation lives with the callers (GameService, EnemyAI); Floor itself is a passive container"*), and `Player`/`Enemy` say the same. Switching to immutable-with-deep-copy here would (a) break consistency with the rest of the domain and (b) pay a full 80×50 tile-grid copy cost per turn for no concrete benefit in v1. The return value is **not** "a new Dungeon" — it is a `TurnResult` carrying the event list and game-over flag (see the domain-events decision below). Mutation lives on the inputs; narrative lives on the result.
- [x] Turn order — all player actions then all enemy AI, or interleaved? → **Interleaved.** Order is: player acts (one action), then for each enemy in deterministic order (sorted by `enemy_id`) the enemy reads the current state (including state changes from enemies that already moved this turn) and acts. Standard roguelike turn model. Deterministic enemy order matters for replay reproducibility — `sorted(floor.enemies, key=lambda e: e.enemy_id)` keeps it stable across runs.
- [x] RNG injection — how is per-turn randomness passed in without `GameService` depending on global `random`? → **Per-turn seeded RNG derived from `(dungeon.seed, dungeon.current_floor_index, dungeon.turn_count)`.** `process_turn` constructs `rng = random.Random(f"{seed}|{floor_index}|{turn_count}")` internally; no caller-side RNG state to thread. Mirrors the `DungeonGenerator` pattern exactly. Requires adding `turn_count: int = 0` to `Dungeon` (small 1.6 model extension) which `process_turn` increments at the end of the turn. Replay benefit: given the seed and the action log, every turn's RNG state is recoverable from `(seed, floor_index, turn_count)` alone — no `random.Random` instance to serialise into the run record. Tests use real seeds; no `unittest.mock` needed for RNG.
- [x] Enemy wake-state storage — where does the per-floor "this enemy is awake" flag live? → **`awake: bool = False` field on `Enemy`.** Smallest, most natural data home — the state is per-enemy semantic ("this goblin has noticed the player"), `Enemy` is already mutable and already persists across turns, and on floor descent the new floor's fresh enemies reset the flag automatically. `GameService` reads `enemy.awake`, calls `enemy_ai.decide_action(..., awake=enemy.awake)`, writes back `enemy.awake = new_awake`. Rejected alternative: a separate `set[UUID]` on `Floor`/`Dungeon` — adds serialisation surface, requires stale-ID cleanup on enemy death, and pays a hash lookup per AI call for no offsetting gain.
- [x] Damage variance — flat damage or RNG variance per hit? → **RNG variance per hit.** Formula: `damage = max(1, attacker.attack + rng.randint(-1, 1) - defender.defense)`. Floor of 1 prevents armor-stacking "0 damage forever" lockouts. Variance ±1 on attack=3 / defense=1 gives a 1–3 hit range — meaningful but not chaotic; predictable enough that the player can plan. Uses the per-turn seeded RNG decided above, so damage rolls are fully reproducible. Specialised variance (boss tells, ranged crits) can layer in later without touching this rule.
- [x] Domain events — does `process_turn` return events alongside mutating state? → **Yes — `TurnResult(events: list[TurnEvent], game_over: bool)`.** `TurnEvent` is a frozen-dataclass union mirroring `Action`'s shape. The v1 surface is: `PlayerMoved`, `PlayerAttacked`, `EnemyAttacked`, `PlayerDamaged`, `EnemyKilled`, `PlayerDied`, `FloorDescended`, `RunAbandoned`, `ActionRejected(reason: str)`. `ItemPickedUp` / `DoorOpened` are deliberately not in v1 — they belong to the `PickUp` and `Open` action paths, which `process_turn` rejects as `not_implemented_v1` until the backing player-inventory and open-door-state models ship. They slot in additively when those land. Why events over direct side effects: (1) the WebSocket layer (task 3.9) serialises events to clients without coupling to `GameService` internals — clean separation of "what happened" from "tell the client"; (2) the replay log *is* the event log — replay = re-emit events from the log; (3) the use-case layer dispatches on events to trigger Celery side effects (leaderboard recalc fires on `PlayerDied`) without `GameService` knowing Celery exists; (4) tests assert on the event list, which is more expressive than checking mutated state diff. `process_turn` still mutates `dungeon`/`player`/`floor` in place — events are the narrative, mutation is the state. Worth a `DECISIONS.md` ADR entry — this is a load-bearing call for Phases 2–3.
- [x] Damage-taken accumulator home (deferred from task 1.7) — where does the cumulative damage counter for the score-penalty formula live? → **`damage_taken: int = 0` on `Player`.** The 1.7 decision left this open ("likely Player or Dungeon — exact home TBD during implementation") and `process_turn` is the implementation point. Player is the right owner: the damage happens *to the player*, the player carries it through the run, and `ScoreService` (task 1.18) reads it once at scoring time. `process_turn` increments it inside the `PlayerDamaged` event emit.
- [x] Does `process_turn` take any port/repository dependencies (per BOARD.md 1.17 "use fake repos")? → **No.** `process_turn(dungeon, player, action) -> TurnResult` is a pure module function (modulo input mutation) — no `IGameRepository`, no `ICachePort`, no constructor at all. Repository orchestration lives one layer up in the planned `ProcessTurn` use case (task 3.2): load `Dungeon` from `IGameRepository`, call `process_turn`, persist on `game_over` / `FloorDescended`, snapshot to `ICachePort` otherwise. Domain tests therefore need no fakes — they instantiate `Dungeon` / `Player` / `Floor` directly via the existing `_dungeon` / `_player` / `_floor_from_grid` helpers in `tests/unit/domain/services/test_game_service.py`. **`QUIZZES.md` task 1.17 Q3 ("FakeGameRepository") encodes the older design and needs an update when the Phase 1 summary quiz is graded** — the correct v1 answer is "no fake is needed because the domain service depends on no ports."

## Phase 3 — Application use cases + API

- [x] Rate limiting — none, in-process (slowapi), or Redis-backed? (task 3.4) → **None for v1.** No external consumers yet; add `slowapi` in Phase 6 when real infra lands — it's a one-file change.
- [x] CORS allowed origins — `localhost:5173` for dev; prod domain TBD. (task 3.4) → **`["http://localhost:5173"]` hardcoded for dev; read from a `CORS_ORIGINS` env var (comma-separated) for prod.** Twelve-factor pattern, consistent with existing env config approach.
- [x] Leaderboard pagination — cursor or offset/limit? Page size? (tasks 3.10–3.12) → **Offset/limit, fixed page size 100, no cursor for v1.** Board frames these as "top 100" endpoints; the dataset is bounded and Redis-cache-served, so offset deep-pagination cost never materialises. Params `?limit=100&offset=0`, `limit` capped at 100. Cursor pagination is additive on a new param if full ranking browse is ever added.
- [x] WebSocket auth — JWT in query string, `Sec-WebSocket-Protocol`, or first client message? (task 3.9) → **JWT in the first client message.** Query-string tokens leak into access/proxy logs and browser history; `Sec-WebSocket-Protocol` abuses a negotiation header and is awkward with the browser `WebSocket` API. Accept the connection, expect an `{"type":"auth","token":"..."}` frame within a short deadline, validate via the existing `get_current_user` logic (task 2.10), close with 1008 (policy violation) if missing/invalid. Keeps the token out of logs and reuses JWT verification.
- [x] Error response shape — FastAPI default, or RFC 7807 Problem Details? (task 3.13) → **RFC 7807 Problem Details (`application/problem+json`).** 3.13 is the schema-design task and this is a portfolio piece; 7807 is the documented HTTP-error standard (`type`, `title`, `status`, `detail`, `instance`), self-describing for the React client, swappable via a single FastAPI custom exception handler. Retrofit the 3.6/3.7/3.8 `HTTPException`s at 3.13 — small and mechanical.
- [x] API versioning — `/v1/` prefix from the start, or add later? (task 3.4) → **`/v1/` prefix from the start.** Single `prefix="/v1"` on the router, zero ongoing cost, avoids a breaking change if any client ever caches the unversioned path.
- [x] `SubmitScore` — sync-persist then enqueue recalc, or enqueue both? (task 3.3) → **Sync-persist then enqueue recalc.** Within the use case's own transaction (UoW): synchronously write the `Score` row via `IScoreRepository` (the repo flushes/merges but does **not** commit — the request-scoped Unit of Work owns the commit, per ADR-0006), **then** enqueue a Celery `score_recalc` task for the async leaderboard-cache rebuild. The score is the user's run result and must not be lost to a dropped message / down worker, so it gets a transactional DB write and a real success/failure response; the leaderboard recalc is derived cache work and is eventually-consistent by nature (matches CLAUDE.md's `score_recalc` = "Async leaderboard rebuild (non-blocking)"). Hexagonal fit: use case owns the transaction boundary (ADR-0006), persists through the port, then fires the task — no domain/app coupling to Celery beyond a task-enqueue port. Trade-off accepted: one synchronous DB round-trip before returning, in exchange for data safety on the one write that must not be lost.

## Phase 4 — Celery workers

- [x] Task retry policy — default exponential backoff, or custom per-task? (tasks 4.2–4.4) → **Uniform default exponential backoff with a cap, applied across 4.2–4.4** (not custom per-task): `autoretry_for=(Exception,)`, `retry_backoff=True`, `retry_backoff_max=600`, `max_retries=3`, `retry_jitter=True`. These tasks are idempotent/derived (per the log-and-drop decision above), so uniform retry-then-drop is the natural fit; per-task tuning can layer in later if a specific task needs it.
- [x] Dead-letter / failed-task handling — Sentry, dedicated queue, or log-and-drop? (task 4.1) → **Log-and-drop via structlog for v1; Sentry deferred to Phase 6.** A terminal task failure emits a structured `structlog` error (task name, task id, call args/kwargs, retry count, exception) and is dropped. Rationale: (1) the Redis broker transport has no native dead-letter routing (that's an AMQP/RabbitMQ feature), so a real DLQ would mean hand-rolling a failed-task Redis list — unjustified infra for v1; (2) these tasks are derived/idempotent, not source-of-truth — the `Score` row is persisted synchronously *before* `score_recalc` is enqueued (see 3.3), so a dropped recalc only means a momentarily stale leaderboard cache that the next game-over rebuilds, and a missed `weekly_leaderboard_reset` is caught the following Monday; (3) log-and-drop matches the "no print, use structlog" convention and gives an observability hook to later point at Sentry. Upgrade path: wiring `sentry-sdk`'s Celery integration into the same failure handler in Phase 6 is additive — logged as a Phase 6 follow-up, not a v1 dependency.
- [x] `map_generation` output — fully rendered `Floor` pushed to Redis, or just the seed? (task 4.3) → **Fully-rendered `Floor` cached, not the seed.** The task's purpose is to keep heavy BSP gen off the turn/request path; since `DungeonGenerator` is a pure seeded function, caching just the seed offloads nothing (the consumer would still run BSP on descent). Cache the computed geometry: serialise the `Floor` in the **application layer** mirroring the `game_state.py` pattern (`serialize_*` + `*_cache_key` + a TTL constant) — never in the cache adapter, which stays a generic `str` store. Deterministic per-floor key `floor:{dungeon_id}:{floor_index}` with a TTL. `ProcessTurn`'s descend path reads **cache-first, falls back to inline generation on miss**, so a dropped/late task only costs one inline gen rather than breaking the descent (consistent with the 4.1 log-and-drop posture).
- [x] Weekly reset archive — snapshot to DB table, to S3/Supabase Storage, or both? (task 4.4) → **Dedicated Postgres table, not Storage, not both.** Leaderboard data is already relational and lives in Postgres (see "Score/leaderboard stays relational separately"); a weekly archive is just the rows that were the weekly top-N at reset time — keep it queryable via the same SQLAlchemy `Base` + Alembic migration + UoW/commit discipline (ADR-0006), zero new infra. Supabase Storage is already scoped to `saves`/`avatars` blobs (task 2.11), not structured records. "Both" is premature — doubles the write/failure surface for no stated v1 need, against the project's "no premature infra" posture. Concretely (refined at implementation against the shipped schema): the weekly board is **window-derived** — `top_n(.., WEEKLY)` filters the *shared* `scores` table by `computed_at >= current_week_start()`, and the global all-time board reads the same rows unwindowed. So "wipe" is **not** a DELETE (that would destroy global history — `models.py`: scores outlive runs); the weekly view already resets itself when the Monday boundary advances. The task therefore: (1) **archives** the *just-completed* week's ranked top-100 — window `[prev Monday 00:00 UTC, this Monday 00:00 UTC)`, reaching one week back so the Beat-boundary run captures the closing week, not the new empty one — into `weekly_leaderboard_archive` (keyed by `week_start`, idempotent delete-then-insert), then (2) **resets the view** by refreshing the `leaderboard:WEEKLY` cache slice to the new week. Non-destructive; one UoW; a thin Celery adapter over the `ResetWeeklyLeaderboard` use case, ports only (mirrors `score_recalc`). Admin op lives on a **new `IScoreAdminRepository`** port, per the existing `IScoreRepository` Q4 doctrine — not a widening of the read port. `score_recalc`-vs-reset race on the WEEKLY key is documented + accepted (self-healing cache), not locked in v1. Upgrade path to off-platform export (Storage/S3) stays additive.

## Cross-cutting

- [x] `structlog` output — JSON to stdout only, or also ship to a collector? → **JSON to stdout only for v1.** Twelve-factor compliant: the runtime emits structured logs to stdout, the *platform* (Docker / ECS / journald) decides where they go. This means swapping the collector — CloudWatch Logs in Phase 6, Loki/Grafana later, Datadog if the project ever needs it — is a *deploy-config* change with **zero application code changes**. Pre-wiring a specific collector now would couple the domain/adapter layers to a transport choice that's almost certain to change, and would make local dev (where you just want `docker compose logs -f api`) painfully indirect. Decision is reversible — the moment a collector is needed, add a sidecar or processor.
- [x] Feature flags — none planned, or bring Unleash / GrowthBook / similar? → **None for v1.** This is a single-developer portfolio project: no rollout cohorts, no A/B testing, no multi-tenant gating, no need to dark-launch a feature behind a kill switch. Adding a flag system before there's anything to flag is premature infrastructure — it adds a dependency, a service to run locally, an environment variable surface, and a layer of indirection on every gated branch, in exchange for zero current value. Re-evaluate **only** if a concrete need surfaces (e.g. shipping a major rule change without breaking existing leaderboard runs) — and even then, the simplest flag system is a Postgres table + a `feature_enabled(name)` adapter, not a full SaaS.
- [x] `import-linter` gate (BOARD.md backlog) — schedule before Phase 2 or after? → **Before Phase 2.** Phase 2 is the first moment a hexagonal violation can plausibly slip in: adapters land (`PostgresGameRepository`, `RedisCache`), and the temptation to import them upward — or to leak `sqlalchemy` types into a domain dataclass field — becomes real. Adding the gate **at the Phase 1 → Phase 2 boundary** catches the first risky PR rather than letting violations accumulate and audit-debt build up. The setup cost is small (one config file + one CI step), and it codifies the architectural rule into a hard CI failure rather than a code-review ask. Do this as the first task of Phase 2 (or a Phase 1.5 chore). Retrofitting later means reading every existing import to confirm clean state, which is the same work plus the risk of having to refactor.
- [x] License for the repo — MIT, Apache-2.0, proprietary? → **MIT.** Simplest permissive license, zero ambiguity, standard for portfolio projects. Matches the framing in `CLAUDE.md` ("HexCrawl is a vehicle for learning… portfolio centrepiece") — the goal is for others to *be able* to read, fork, learn from, and reuse this code. **Not Apache-2.0** because there are no patents to grant and the extra notice requirements add friction with no benefit at this scale. **Not proprietary** because closed-source hostile-licensing actively undermines the "show this off, let people learn from it" goal. Add `LICENSE` file at repo root with the standard MIT text + copyright line. Reversible later if the project ever pivots to commercial use, but the pivot is unlikely.
- [x] Per-turn CPU budget — at what wall-clock cost does a `process_turn` step stop being "inline on the event loop" and become a Celery candidate? `DECISIONS.md` ADR-0003 currently uses a placeholder of ~5 ms as the threshold; needs to be substantiated (or replaced) by real profiling once the WS turn loop is wired up. Open questions inside this: do we measure p50, p95, or worst-case? Per turn or per session-second? Is the budget different for boss floors / floor 10+ where BSP regen may be more frequent? Decision should land before Phase 4 (`map_generation` task), because that's the first time we'll concretely choose what to push off the loop. → **Don't fabricate a substantiated number for 4.3; use a depth heuristic and keep ~5 ms as an explicit placeholder.** The ADR-0003 ~5 ms figure is a made-up placeholder — no profiling exists yet (the WS turn loop only landed in Phase 3). For 4.3 the trigger that actually needs to land is narrower than a wall-clock budget: the board already says "floors 10+", so the offload trigger is **depth-based — pre-gen via `map_generation` when `floor_index >= 10`**, not a measured threshold. The ~5 ms in ADR-0003 stays flagged as a placeholder; when profiling does happen, **measure p95 per turn** (worst-case is too noisy, p50 hides the tail). Unblocks 4.3 without inventing profiling data.
